// Foundation

What is Query Processing?

Query processing is the bridge between what you ask (high-level declarative SQL) and how the database actually retrieves data (low-level procedural operations).

🗣️
beginner

Query Language

SQL is a tuple calculus language. Users write what they want — the processor figures out how to get it.

Tuple Calculus: { t | F(t) } Where t = tuple variable F(t) = well-formed formula Domain Calculus: { x,y | EMP(x,y,"Manager") } → position-sensitive variables
⚙️
beginner

Query Processor Role

Transforms declarative queries into optimal procedural operations. Handles three core jobs:

1. Query Language Support (SQL) 2. Execution Methodology → parsing → planning → running 3. Query Optimization → find the "best" plan
🌐
intermediate

Distributed Extra Challenges

When data lives on multiple sites, the processor must also handle:

+ Query Decomposition → break query into sub-queries + Data Localization → find which site has which data + Communication Cost Optimization → minimize network traffic

// Level 01

🟢 Beginner

Master the fundamentals: centralized vs distributed query processing, basic optimization, and the cost problem.

CENTRALIZED vs DISTRIBUTED PLANS

Example Query:
SELECT ENAME FROM EMP E, ASG G WHERE E.ENO = G.ENO AND TITLE = "manager"

Relations EMP and ASG are fragmented across multiple sites. The processor must choose how to execute this.

📦 Plan I — Brute Force

Transport ALL fragments to one query site, execute everything there. Simple to implement but causes massive network traffic.

Result = (EMP1 ∪ EMP2) ⋈ENO σTITLE="manager"(ASG1 ∪ ASG2) [All data → Site 5 → join]

Total Cost: 23,000 units 💀

⚡ Plan II — Optimized

Apply selection first at local sites, transfer only matching tuples, then join small results at query site.

ASG1' = σTITLE="manager"(ASG1) ASG2' = σTITLE="manager"(ASG2) EMP1' = EMP1 ⋈ENO ASG1' EMP2' = EMP2 ⋈ENO ASG2' Result = (EMP1' ∪ EMP2')

Total Cost: 460 units ✅

COST BREAKDOWN — ASSUMPTIONS

Given: size(EMP)=400, size(ASG)=1000, 20 tuples with TITLE="manager"
Costs: tuple access = 1 unit · tuple transfer = 10 units
Clustering: ASG clustered on TITLE, EMP clustered on ENO
Operation Plan I Plan II Winner
Transfer EMP to query site 400 × 10 = 4,000 — Plan II
Transfer ASG to query site 1000 × 10 = 10,000 — Plan II
Produce ASG' (selection) 1000 × 1 = 1,000 (10+10) × 1 = 20 Plan II
Join EMP and ASG' 400 × 20 × 1 = 8,000 (10+10) × 1 × 2 = 40 Plan II
Transfer ASG' to EMP sites — (10+10) × 10 = 200 —
Transfer EMP' to result site — (10+10) × 10 = 200 —
TOTAL 23,000 460 Plan II — 50× faster!

NETWORK TOPOLOGY IMPACT

🌍
WAN

Wide Area Network

Communication cost dominates everything. Low bandwidth + high protocol overhead. Algorithm goal: minimize data transfer at all costs.

Optimize for: min(communication cost) Ignore: CPU cost, I/O cost Use: global schedule → minimize transfer
🏢
LAN

Local Area Network

Communication cost less dominant. Consider total cost function. Broadcasting can be exploited. Special star-network algorithms available.

Optimize for: min(I/O + CPU + comm) Exploit: broadcast capabilities Use: total cost function
// Level 02

🟡 Intermediate

Deep dive into Step 1: Query Decomposition — from calculus query to optimized relational algebra.

Step 1

Normalization

Convert query to CNF or DNF. Lexical + syntactic analysis. Type checking on qualifications. Maps OR → union, AND → join/selection.

Step 2

Analysis

Reject type-incorrect or semantically incorrect queries. Use query graph — if NOT connected, query is semantically incorrect.

Step 3

Simplification

Eliminate redundant predicates using idempotency rules. Reduce complex boolean expressions to minimal form.

Step 4

Restructuring

Translate calculus query to optimal relational algebra. Build query tree. Apply transformation rules to improve performance.

CNF vs DNF FORMS

∧
CNF

Conjunctive Normal Form

AND of clauses, each clause is OR of predicates.

(p11 ∨ p12 ∨ ... ∨ p1n) ∧ ... ∧ (pm1 ∨ pm2 ∨ ... ∨ pmn)
∨
DNF

Disjunctive Normal Form

OR of clauses, each clause is AND of predicates.

(p11 ∧ p12 ∧ ... ∧ p1n) ∨ ... ∨ (pm1 ∧ pm2 ∧ ... ∧ pmn)

NORMALIZATION EXAMPLE

SELECT ENAME FROM EMP, ASG WHERE EMP.ENO = ASG.ENO AND ASG.JNO = "J1" AND (DUR = 12 OR DUR = 24)
→ CNF form:
(EMP.ENO = ASG.ENO) ∧
(ASG.JNO = "J1") ∧
(DUR = 12 ∨ DUR = 24)

ANALYSIS — QUERY GRAPH

Rule: A conjunctive query (no negation) is semantically incorrect if its query graph is NOT connected.

Query graph has: one node per relation + edges for joins and projections. All nodes must be reachable from the result node.
🔗
Type Incorrect

Type Error Examples

-- Undefined attribute: SELECT E# FROM EMP -- E# doesn't exist -- Type mismatch: WHERE ENAME > 200 -- string > number
🕸️
Semantically Incorrect

Disconnected Query Graph

If you have 3 relations but no join condition links them all — some relation floats disconnected → the query produces meaningless Cartesian product.

EMP ──⋈── ASG PROJ (isolated!) ↑ connected ↑ PROJ must also connect here

SIMPLIFICATION EXAMPLE

Complex query with redundant predicates:
SELECT TITLE FROM EMP WHERE (NOT(TITLE="Programmer") AND (TITLE="Programmer" OR TITLE="Electrical Eng.") AND NOT(TITLE="Electrical Eng.")) OR ENAME="J.Doe"
-- DNF Simplification:
p1 = TITLE="Programmer", p2 = TITLE="Elec.Engr", p3 = ENAME="J.Doe"
 
(¬p1 ∧ p1 ∧ ¬p2) ∨ (¬p1 ∧ p2 ∧ ¬p2) ∨ p3
→ (false ∧ ¬p2) ∨ (¬p1 ∧ false) ∨ p3
→ false ∨ false ∨ p3
→ p3 (ENAME = "J.Doe")
 
-- Simplified query:
SELECT TITLE FROM EMP WHERE ENAME = "J.Doe"

RESTRUCTURING — TRANSFORMATION RULES

⟺ Commutativity of Binary Operations

+
Joins and Unions can swap order:
R × S ⟺ S × R
R ∪ S ⟺ S ∪ R
R ⋈ S ⟺ S ⋈ R

Use this to put the smaller relation first in joins — reduces intermediate result size.

⟺ Associativity of Binary Operations

+
Reorder joins freely:
(R × S) × T ⟺ R × (S × T)
(R ⋈ S) ⋈ T ⟺ R ⋈ (S ⋈ T)

Use this to find the join order that minimizes intermediate sizes.

⟺ Idempotence of Unary Operations

+
Group multiple projections:
Π_A'(Π_A''(R)) ⟺ Π_A'(R) [if A' ⊆ A'']

Group multiple selections:
σ_p1(σ_p2(R)) ⟺ σ_p1∧p2(R)

Access relation once instead of multiple times!

⟺ Commuting Selection with Binary Ops

+
Push selections down the tree (most important optimization!):
σ_p(Ai)(R × S) ⟺ (σ_p(Ai)(R)) × S
σ_p(Ai)(R ⋈ S) ⟺ (σ_p(Ai)(R)) ⋈ S
σ_p(Ai)(R ∪ S) ⟺ σ_p(Ai)(R) ∪ σ_p(Ai)(S)

Why? Applying selection first reduces relation size before expensive join/product operations.

⟺ Commuting Projection with Binary Ops

+
Push projections down similarly:
Π_C(R × S) ⟺ Π_A(R) × Π_B(S) [C = A ∪ B]
Π_C(R ⋈ S) ⟺ Π_C(R) ⋈ Π_C(S)
Π_C(R ∪ S) ⟺ Π_C(R) ∪ Π_C(S)

Eliminate unneeded columns early to reduce data volume.

🔴 Advanced

Step 2: Data Localization — translating algebra queries on global relations into queries on physical fragments, then reduction and global optimization.

FRAGMENTATION TYPES

↔️
PHF

Primary Horizontal Fragmentation

Relation split by rows using selection predicates. Each fragment has a subset of tuples satisfying a condition.

EMP1 = σ(ENO ≤ "E3")(EMP) EMP2 = σ("E3" < ENO ≤ "E6")(EMP) EMP3 = σ(ENO > "E6")(EMP) EMP = EMP1 ∪ EMP2 ∪ EMP3
↕️
VF

Vertical Fragmentation

Relation split by columns using projection. Each fragment has a subset of attributes (columns).

EMP1 = Π(ENO, ENAME)(EMP) EMP2 = Π(ENO, TITLE)(EMP) Note: ENO must be in both → needed for reconstruction
🔀
DHF

Derived Horizontal Fragmentation

Fragment defined by joining with another relation. ASG fragmented based on EMP's fragmentation criterion.

EMP1 = σTITLE="Programmer"(EMP) EMP2 = σTITLE≠"Programmer"(EMP) ASG1 = ASG ⋈ENO EMP1 ASG2 = ASG ⋈ENO EMP2
🧬
HF

Hybrid Fragmentation

Combines horizontal + vertical fragmentation. Both row and column splits applied.

EMP1 = σ(ENO≤"E4")(Π(ENO,ENAME)(EMP)) EMP2 = σ(ENO>"E4")(Π(ENO,ENAME)(EMP)) EMP3 = Π(ENO,TITLE)(EMP)

REDUCTION STRATEGIES

🔴 Reduction with Selection (PHF)

+
Rule: σ_pj(Rj) = ∅ if for all x in R: ¬(pi(x) ∧ pj(x))

If the selection predicate contradicts the fragment's definition predicate → the result is empty → eliminate that fragment.

Example:
Query: SELECT * FROM EMP WHERE ENO = "E5"
EMP1 = σ(ENO ≤ "E3")(EMP) → ENO="E5" contradicts ENO≤"E3" → EMP1 empty, skip
EMP2 = σ("E3" < ENO ≤ "E6")(EMP) → ENO="E5" satisfies this → keep EMP2 only
EMP3 = σ(ENO > "E6")(EMP) → contradicts → skip

🔴 Reduction with Join (PHF)

+
Steps:
1. Distribute join over union: (R1 ∪ R2) ⋈ S ⟺ (R1 ⋈ S) ∪ (R2 ⋈ S)
2. Eliminate empty joins: Ri ⋈ Rj = ∅ if their defining predicates are mutually exclusive

Example:
EMP fragmented by ENO ≤ "E3" / ENO > "E3"
ASG fragmented by ENO ≤ "E3" / ENO > "E3"
After distributing: 4 potential join pairs
EMP1 ⋈ ASG2 = ∅ (ENO ≤ "E3" and ENO > "E3" can't match)
EMP2 ⋈ ASG1 = ∅ (same contradiction)
Result: only EMP1⋈ASG1 and EMP2⋈ASG2 survive!

🔴 Reduction for Vertical Fragmentation (VF)

+
Rule: Π_D(Ri) is useless if projection attributes D are not in A' (Ri's attribute set)

Example:
EMP1 = Π(ENO, ENAME)(EMP) — has ENAME
EMP2 = Π(ENO, TITLE)(EMP) — has TITLE but no ENAME

Query: SELECT ENAME FROM EMP
→ Π_ENAME applied to EMP2 is useless (ENAME not in EMP2)
→ Eliminate EMP2 from the query, keep only EMP1

🔴 Reduction for Derived HF (DHF)

+
Process:
1. Apply selection first (push down σ)
2. Distribute joins over unions
3. Apply PHF join reduction rules to eliminate empty results

Example Query:
WHERE ASG.ENO = EMP.ENO AND EMP.TITLE = "Mech. Eng."
After applying σ TITLE="Mech.Eng." first:
σ on EMP1(TITLE="Programmer") = ∅ → ASG1 ⋈ EMP1 branch eliminated
Only ASG2 ⋈ EMP2(TITLE="Mech.Eng.") survives

🔴 Reduction for Hybrid Fragmentation

+
Combine all rules:
1. Remove empty relations (contradicting σ on horizontal fragments)
2. Remove useless relations (Π on vertical fragments with wrong attributes)
3. Distribute joins over unions → isolate and remove useless joins

Example (Hybrid):
EMP1 = σ(ENO≤"E4")(Π(ENO,ENAME)(EMP))
EMP2 = σ(ENO>"E4")(Π(ENO,ENAME)(EMP))
EMP3 = Π(ENO,TITLE)(EMP)
Query: SELECT ENAME FROM EMP WHERE ENO = "E5"
Step 1: σ ENO="E5" on EMP1 → contradicts ENO≤"E4" → EMP1 eliminated
Step 2: Π_ENAME on EMP3 → ENAME not in EMP3 → EMP3 eliminated
Result: only EMP2 survives → Π_ENAME(σ_ENO="E5"(EMP2))

GLOBAL OPTIMIZATION

📊
Cost Function

Optimization Objective

Minimize the combined cost function across all factors:

Cost = I/O cost + CPU cost + Communication cost → WAN: weights heavily comm cost → LAN: balanced weights → Also: maximize throughput
🔍
Search Strategy

Exhaustive vs Heuristic

Exhaustive: Cost-based, finds optimal solution. Combinatorial complexity — only for small spaces.

Heuristic: Not optimal but practical. Key heuristics:

  • Re-group common sub-expressions
  • Perform σ and Π first always
  • Replace join with semijoins
  • Reorder ops to reduce intermediate sizes
  • Optimize individual operations
⏱️
Timing

Static vs Dynamic vs Hybrid

STATIC (compile time): + amortized over many executions - hard to estimate intermediate sizes - appropriate for exhaustive search DYNAMIC (run time): + accurate intermediate sizes - expensive, repeated every execution HYBRID: 1. Compile with static algorithm 2. If estimate error > threshold → re-optimize at runtime
📈
Statistics Needed

Required Statistical Data

  • Relation cardinality (tuple count)
  • Size of a tuple
  • Fraction participating in join
  • Attribute domain cardinality
  • Number of distinct values
Assumptions: • Attribute values: independent • Distribution: uniform within domain

DECISION SITES FOR OPTIMIZATION

🎯
Centralized

Single Site Decision

One site determines the entire schedule. Simple approach but requires knowledge about all distributed data.

🤝
Distributed

All Sites Cooperate

All sites coordinate to determine optimal schedule using only local information. More complex but more scalable.

⭐
Hybrid

One Site + Local Decisions

One site determines global schedule. Each participating site optimizes its own local sub-queries independently.

SEMIJOINS — REDUCING TRANSFER COST

Idea: Instead of transferring full relation R to join with S at a remote site, send only the join attribute of R first, filter S, then send back only matching tuples.

R ⋉ S = Π_A(R ⋈ S) — semijoin of R with S on A

Effective when: the join selectivity is low (few matching tuples) and communication overhead is not significant relative to savings.
-- Without semijoin (naive):
Site 1 → transfer full R → Site 2
Site 2 → R ⋈ S → result
 
-- With semijoin:
Site 1 → send Π_A(R) [join attributes only] → Site 2
Site 2 → compute S' = S ⋉ Π_A(R) [filter S] → send S' back
Site 1 → R ⋈ S' [small S' only!] → result

OPTIMIZATION GRANULARITY

Single Query at a Time

Each query optimized independently.
Cannot reuse common intermediate results.

Multiple Queries at a Time

Batch queries together, share common sub-expressions.
Efficient when many similar queries run together.
Decision space much larger.

4 Steps of Decomposition

Fragmentation Rules

  • Π_D(Ri) useless if D ⊄ A' → eliminate fragment
  • Distribute joins over unions, then apply PHF rules
  • Combine all three reduction rules

Must Know

Key Transformation Rules

σ pushed DOWN before joins ✓ Π pushed DOWN before joins ✓ σ(σ(R)) → σ(p1∧p2)(R) ✓ Π(Π(R)) → Π_A'(R) ✓ Commutativity: R⋈S = S⋈R ✓ Associativity: (R⋈S)⋈T = R⋈(S⋈T) ✓
💰
Must Know

Optimization Dimensions

  • Granularity: single query vs multiple queries
  • Timing: static / dynamic / hybrid
  • Decision site: centralized / distributed / hybrid
  • Search: exhaustive (optimal) / heuristic (practical)
  • Network: WAN (min comm) / LAN (total cost)

Must Know

Semijoin Benefit

R ⋉ S = Π_A(R ⋈ S) Use when: • Join selectivity is LOW • Communication overhead LOW • Many non-matching tuples in R Avoid when: • Most tuples match anyway • Small relations → just send all

Statistics Assumptions

  • Attribute values are independent of each other
  • Values are uniformly distributed within their domain
  • These assumptions can cause estimation errors → justify hybrid timing
